# Мини-анкета — Full-Stack приложение

> Домашнее задание: генерация простого приложения с помощью AI-ассистента в IDE

---

## Цель

- Создать минимальное full-stack приложение (backend + frontend)
- Настроить API для получения конфигурации анкеты и отправки ответов
- Развернуть фронтенд, подключённый к backend API
- Реализовать хранение пользовательских ответов в памяти backend-приложения
- Проверить работу приложения локально

---

## Описание задания

Студенту необходимо сгенерировать и собрать простое приложение **«Мини-анкета»**, состоящее из backend-части и небольшого frontend-интерфейса.

---

## Требуемый функционал

### Backend API

| Метод | Эндпоинт | Описание |
|-------|----------|----------|
| `GET` | `/questions` | Возвращает список вопросов анкеты (3–5 шт., жёстко заданных) |
| `POST` | `/answers` | Принимает ответы пользователя и сохраняет их в памяти (`slice` / `map`) |

### Frontend

- Загружает вопросы с backend
- Отображает их в HTML/JS интерфейсе
- Отправляет заполненные ответы через `POST /answers`
- Показывает пользователю сообщение **«Спасибо!»** после отправки

---

## Технологии

| Слой | Варианты |
|------|----------|
| **Backend** | Go |
| **Frontend** | Чистый HTML + JS, React, Vue |

# Claude Code — Workflow

```

│       ├── plan.md             # Этап 3: план реализации по фазам
│       └── implement.md        # Этап 4: кодирование и review

```

---

## Workflow

| Этап | Скилл | Артефакт | Действие |
|------|-------|----------|----------|
| 1 | `/research` | `docs/research.md` | Изучить задачу, задать вопросы |
| 2 | `/design`   | `docs/design.md`   | Спроектировать архитектуру |
| 3 | `/plan`     | `docs/plan.md`     | Разбить на фазы |
| 4 | `/implement`| код + коммит       | Реализовать фазу, запустить тесты |

---

## Этап 1 — Research

**Запуск команды:** `/research`

Анализ структуры, зависимостей, рисков; уточняющие вопросы до старта

Выход: `docs/research.md` со структурой:
```
# Research: <название>
## Тип задачи · Контекст · Понимание задачи
## Риски и неизвестные · Уточняющие вопросы · Предположения
## [после ответов] Ответы · Итоговое понимание
```

## Этап 3 — Plan

**Запуск команды:** `/plan`

Разбивка на самодостаточные фазы. Критерии одной фазы:

- не более 1–2 файлов / пакетов
- не более ~200–300 строк нового кода
- один интерфейс или один endpoint
- тесты в одном `_test.go`
- один `git diff`, читаемый за 5 минут

Формат фазы:
```
# Phase N: <название>
## Цель · Файлы · Тесты · Критерий Done
```

Второй режим: **review плана** — проверка как опытный Go-разработчик (edge cases, идиоматика, конкурентность).

---

## Этап 4 — Implement

**Запуск команды:** `/implement`

Правила кодирования:

1. Тесты **до** реализации (TDD)
2. Коммит после каждой завершённой фазы (Conventional Commits)
3. После реализации: `make test && make lint`
4. Все ошибки исправляются **до** перехода к следующей фазе
5. В `plan-<name>.md` фаза отмечается выполненной

Переход к следующей фазе — только с подтверждением.

Второй режим: **code review** — свежий взгляд на реализованную фазу.
